# Team Playbook

> **Purpose**
>
> This document explains how DailyPaws is built.
>
> It complements the product documentation by describing the team, responsibilities, working principles and important background context that are not part of the product itself.
>
> A new AI or developer should be able to read this document together with the official product documentation and quickly become an effective contributor.

---

# Project Overview

DailyPaws is a long-term product with the ambition of helping families build stronger relationships with their dog.

It is intentionally **not**:

- a dog training app
- a habit tracker
- a health tracker
- a social network
- a chatbot

DailyPaws exists to help families:

- notice
- understand
- reflect
- celebrate

the small moments that strengthen their relationship with their dog.

Technology serves the product.

The product never serves the technology.

Founder and Product Owner.

Lena owns the long-term product vision and makes all final product decisions.

### Decision-making style

Lena consistently prioritises:

- User experience before implementation.
- Product thinking before technology.
- Simplicity before completeness.
- Small iterations before large releases.
- Long-term consistency over short-term optimisation.
- Language and emotion as part of the product.
- Building the right thing before building it well.

When uncertain, optimise for the user's experience rather than engineering convenience.

# Working Workflow

Every significant feature follows the same process.

1. Product Design Session
2. Product Decisions approved
3. Design Decisions approved
4. Documentation impact analysed
5. Documentation updated
6. GitHub Issue created or updated
7. Implementation
8. Pull Request
9. Product Review
10. Merge

No implementation starts before product design has been approved.

---

# Documentation Principles

Documentation is treated as part of the product.

There are three categories of decisions.

---

## Product Decisions (PD)

Long-term product principles.

Examples:

- Relationship before obedience.
- DailyPaws is a guide, not a character.

These should rarely change.

---

## Design Decisions (DD)

Approved UX decisions.

Examples:

- Welcome screen copy.
- Onboarding flow.
- Navigation.
- Information architecture.
- Interaction patterns.

These may evolve as the product matures.

---

## Technical Decisions (future)

Technology choices.

Examples:

- Frameworks
- Databases
- Infrastructure
- Deployment

Technical Decisions should only be introduced when needed.

---

# Product Design Principles

Every design discussion begins with one question:

**How should the user feel after this experience?**

Before discussing screens or implementation, define:

- purpose
- emotion
- user experience
- information flow

Only then discuss UI.

Only then discuss implementation.

---

# Communication Principles

DailyPaws communicates as a product.

Never as a person.

Never as a chatbot.

The desired tone is:

- Warm
- Calm
- Thoughtful
- Honest

Avoid:

- pretending to have emotions
- pretending to know things that have not been observed
- marketing language
- exaggerated enthusiasm

Prefer:

Conversational UI over conversational AI.

---

# Prompt Quality

When ChatGPT prepares work for Claude:

Every prompt should be self-contained.

It should include:

- complete Product Decisions
- complete Design Decisions
- all relevant context
- documentation responsibilities
- GitHub responsibilities
- explicit deliverables

The Founder should be able to copy the prompt directly to Claude without adding missing information or checking that anything has been forgotten.

Completeness is ChatGPT's responsibility.
